Lecture 2

Git and GitHub Introduction

১. What is Git? (গিট কী?)

Git হলো একটি Version Control System (VCS)। সহজ কথায়, এটি একটি সফটওয়্যার যা আপনার প্রজেক্টের প্রতিটি ছোট-বড় পরিবর্তনের ইতিহাস (History) রেকর্ড করে রাখে।

  • এটি লিনাক্সের জনক লিনাস টরভাল্ডস তৈরি করেছেন।
  • এটি আপনার নিজের কম্পিউটারে (Offline-এ) কাজ করে।

২. Key Git Concepts (গিটের মূল ধারণা)

গিট বুঝতে হলে নিচের ৩টি বিষয় সবথেকে গুরুত্বপূর্ণ:

  • Repository (রিপোজিটরি): আপনার প্রজেক্টের মূল ফোল্ডার, যেখানে গিট তার সব ডাটাবেস জমা রাখে।
  • Commit (কমিট): প্রজেক্টের একটি নির্দিষ্ট সময়ের "স্ন্যাপশট" বা সেভ পয়েন্ট।
  • Branch (ব্রাঞ্চ): মূল কাজ ঠিক রেখে আলাদাভাবে কোনো নতুন ফিচার ট্রাই করার জন্য একটি আলাদা শাখা।

৩. Working with Git (গিট কীভাবে কাজ করে?)

গিট মূলত ৩টি ধাপে বা "States"-এ কাজ করে:

  • Working Directory: যেখানে আপনি সরাসরি ফাইল এডিট বা ডিলিট করছেন।
  • Staging Area: git add করার পর ফাইলগুলো এখানে আসে। এটি হলো সেভ (Commit) করার জন্য প্রস্তুতির জায়গা।
  • Local Repository: git commit করার পর ফাইলগুলো চিরস্থায়ীভাবে ইতিহাসের অংশ হয়ে যায়।

৪. Why Git? (কেন গিট ব্যবহার করবেন?)

  • টাইম ট্রাভেল: আপনি চাইলে এক বছর আগে আপনার কোড কেমন ছিল, সেখানে ফিরে যেতে পারেন।
  • নিরাপত্তা: ভুল করে কোড ডিলিট হয়ে গেলেও গিট থেকে তা উদ্ধার করা সম্ভব।
  • কোলাবরেশন: একই প্রজেক্টে পৃথিবীর বিভিন্ন প্রান্ত থেকে হাজার হাজার মানুষ একসাথে কাজ করতে পারে।
  • উন্নত ট্র্যাকিং: কে, কখন, ফাইলের কোন লাইনে কী পরিবর্তন করেছে, তা একদম পরিষ্কারভাবে দেখা যায়।

৫. What is GitHub? (গিটহাব কী?)

অনেকে গিট এবং গিটহাবকে এক মনে করেন, কিন্তু তারা আলাদা।

  • গিটহাব হলো একটি অনলাইন ক্লাউড প্ল্যাটফর্ম বা ওয়েবসাইট।
  • এটি আপনার কম্পিউটারের গিট রিপোজিটরিগুলোকে অনলাইনে জমা রাখার জায়গা (Hosting)।
  • সহজ উদাহরণ: সহজ উদাহরণ: গিট যদি হয় আপনার ফোনের "ক্যামেরা" (যা ছবি তোলে বা রেকর্ড করে), তবে গিটহাব হলো "গুগল ড্রাইভ" বা "আইক্লাউড" (যেখানে আপনি ছবিগুলো অনলাইনে ব্যাকআপ রাখেন)।

৬. Git and GitHub Introduction (সমন্বয়)

আপনি যখন আপনার পিসিতে Git ব্যবহার করে কাজ শেষ করেন, তখন সেই কাজগুলো সবার সাথে শেয়ার করার জন্য বা ব্যাকআপ রাখার জন্য GitHub-এ push করে পাঠিয়ে দেন। এর মাধ্যমে অন্যান্য ডেভেলপাররা আপনার কোড দেখতে পারে এবং প্রয়োজনে সাহায্য করতে পারে।

    সংক্ষেপে:
  • Git: টুল বা সফটওয়্যার (লোকাল পিসিতে থাকে)।
  • GitHub: ওয়েবসাইট বা ক্লাউড সার্ভিস (অনলাইনে থাকে)।

Lecture 3

Git Install

১. Git Install এবং How to Install Git

উইন্ডোজে গিট ইন্সটল করার সবথেকে নিরাপদ উপায় হলো অফিসিয়াল ওয়েবসাইট git-scm.com থেকে ফাইল ডাউনলোড করা।

  • ডাউনলোড করা .exe ফাইলটি ওপেন করে সাধারণত সব অপশনে Next দিলেই এটি সুন্দরভাবে ইন্সটল হয়ে যায়। তবে মাঝখানের কিছু টেকনিক্যাল সেটিংস বুঝে নেওয়া ভালো।

২. Git Bash কী?

ইন্সটল করার পর আপনি Git Bash নামে একটি অ্যাপ পাবেন। উইন্ডোজের সাধারণ কমান্ড প্রম্পটের (CMD) চেয়ে এটি অনেক বেশি শক্তিশালী। কারণ এটি আপনাকে উইন্ডোজের ভেতরেই লিনাক্স বা ইউনিক্স টার্মিনালের স্বাদ দেয়। গিটের সব কমান্ড এখানে কোনো ঝামেলা ছাড়াই চলে।

৩. Verifying Your Installation (ইন্সটল হয়েছে কি না দেখা)

ইন্সটল করার পর গিট ঠিকমতো কাজ করছে কি না তা চেক করতে আপনার টার্মিনালে (CMD বা Git Bash) এই কমান্ডটি দিন:

    Bash
    git --version

যদি আপনি git version 2.x.x এর মতো কোনো আউটপুট পান, তবে বুঝবেন গিট সফলভাবে ইন্সটল হয়েছে।

৪. Default Editor

ইন্সটলেশনের সময় গিট আপনাকে জিজ্ঞেস করবে আপনি কোন এডিটর (যেমন: Vim, Notepad++, VS Code) ব্যবহার করতে চান।

  • যদি ভুলে Vim সিলেক্ট হয়ে থাকে, তবে কোনো বড় কমান্ডের সময় গিট একটি কালো স্ক্রিন ওপেন করতে পারে যা থেকে বের হওয়া কঠিন।
  • পরে এটি পরিবর্তন করতে চাইলে লিখুন:
  • git config --global core.editor "notepad" (অথবা আপনার পছন্দের এডিটর)।

৫. PATH Environment (পাথ এনভায়রনমেন্ট)

এটি অত্যন্ত গুরুত্বপূর্ণ। আপনি যদি চান যে আপনার কম্পিউটারের যেকোনো ফোল্ডার থেকে বা যেকোনো টার্মিনাল (যেমন আপনার বর্তমান CMD) থেকে গিট কাজ করুক, তবে গিটকে PATH-এ যুক্ত থাকতে হবে।

How to Add Git to PATH after Installation?

যদি ইন্সটলেশনের সময় এটি না করে থাকেন, তবে ম্যানুয়ালি করতে হবে:

  • উইন্ডোজ সার্চে লিখুন "Edit the system environment variables" ।
  • Environment Variables বাটনে ক্লিক করুন।
  • System Variables এর নিচে Path খুঁজে বের করে Edit ক্লিক করুন।
  • New ক্লিক করে গিটের বিন ফোল্ডারের লোকেশন (যেমন: C:\Program Files\Git\bin) লিখে দিন।
  • সব ওকে (OK) করে দিয়ে আপনার টার্মিনালটি রিস্টার্ট করুন।

৬. Line Endings (লাইন এন্ডিং)

উইন্ডোজ এবং লিনাক্স ফাইল সেভ করার সময় আলাদা ফরম্যাট ব্যবহার করে।

  • ইন্সটলেশনের সময় সব সময় "Checkout Windows-style, commit Unix-style line endings" সিলেক্ট করবেন। এতে আপনার কোড যখন অনলাইনে (GitHub) যাবে, তখন ফরম্যাটিং নষ্ট হবে না।

৮. Troubleshooting & Common Installation Issues

সাধারণ কিছু সমস্যা:
  • 'git' is not recognized: এটি হওয়ার প্রধান কারণ হলো গিট PATH-এ যুক্ত নেই। ওপরের ৫ নম্বর ধাপটি অনুসরণ করলে এটি ঠিক হয়ে যাবে।
  • Permission Denied: অনেক সময় সি-ড্রাইভে কাজ করার সময় পারমিশন চায়। সেক্ষেত্রে আপনার টার্মিনালটি "Run as Administrator" দিয়ে ওপেন করুন।
  • Installation stuck: যদি ইন্সটলেশন আটকে যায়, তবে এন্টিভাইরাস বা ফায়ারওয়াল সাময়িকভাবে বন্ধ করে আবার চেষ্টা করুন।

Lecture 4

Git Config

Git-এ কাজ শুরু করার আগে Git Config সেটআপ করা অনেকটা আপনার ডিজিটাল আইডি কার্ড বা পরিচয়পত্র তৈরি করার মতো। আপনি যখনই কোনো প্রজেক্টে কাজ করবেন, গিট আপনার এই কনফিগারেশন থেকেই জানতে পারবে কে কোডটি লিখেছে।

নিচে আপনার প্রতিটি পয়েন্ট বিস্তারিতভাবে বুঝিয়ে বলা হলো:

১. Git Config কী ও কেন এটি করবেন? (Why Configure Git?)

গিট একটি Version Control System। এর কাজ হলো কে, কখন, কী পরিবর্তন করেছে তার হিসাব রাখা। আপনি যখন একটি 'Commit' করেন, গিট সেই কমিটের সাথে আপনার নাম এবং ইমেইল জুড়ে দেয়।

  • কেন জরুরি: আপনি যদি এটি কনফিগার না করেন, গিট আপনাকে কমিট করতে দেবে না। এটি টিম ওয়ার্কের জন্য অপরিহার্য যাতে বোঝা যায় কোন ডেভেলপার কোন কাজটি করেছে।

২. কনফিগারেশনের তিনটি লেভেল (Configuration Levels)

গিট তিনটি স্তরে সেটিংস জমা রাখে। আপনার প্রয়োজন অনুযায়ী আপনি এগুলো ব্যবহার করতে পারেন:

  • Local (--local): এটি শুধুমাত্র একটি নির্দিষ্ট প্রজেক্ট বা ফোল্ডারের জন্য কাজ করে।
  • Global (--global): এই পিসিতে আপনার (বর্তমান ইউজার) সব প্রজেক্টের জন্য কাজ করবে। এটিই সবথেকে বেশি ব্যবহৃত হয়।
  • System (--system):এই কম্পিউটারের সব ইউজার এবং সব প্রজেক্টের জন্য কাজ করবে।

৩. User Name ও Email Address সেট করা

গিট ইনস্টল করার পর প্রথম কাজ হলো আপনার নাম এবং ইমেইল সেট করা। টার্মিনালে (CMD/Bash) নিচের কমান্ডগুলো লিখুন:

নাম সেট করা:
git config --global user.name "আপনার নাম"

ইমেইল সেট করা:
git config --global user.email"yourname@example.com"

৪. ডিফল্ট ব্রাঞ্চের নাম পরিবর্তন (Default Branch Name)

আগে গিটে প্রথম ব্রাঞ্চের নাম ডিফল্টভাবে master থাকতো। এখন আধুনিক প্রজেক্টে সবাই main ব্যবহার করতে পছন্দ করে। আপনি এটি আগে থেকেই সেট করে রাখতে পারেন:

Bash
git config --global init.defaultBranch main

৫. আপনার কনফিগারেশন দেখা (Viewing Your Configuration)

বর্তমানে কী কী সেটিংস আপনার গিটে আছে তা দেখার জন্য নিচের কমান্ডটি দিন:

Bash
git config --list

এটি আপনার নাম, ইমেইল এবং অন্যান্য সব সেটিংস একটি তালিকায় দেখাবে।

৬. সেটিংস পরিবর্তন বা মুছে ফেলা (Changing or Unsetting)

কখনো যদি ভুল তথ্য দিয়ে থাকেন বা সেটিংস বদলাতে চান, তবে নিচের পদ্ধতিগুলো অনুসরণ করুন:

  • পরিবর্তন করা (Changing): শুধু আগের কমান্ডটি আবার নতুন তথ্য দিয়ে লিখলেই পুরনোটি মুছে নতুনটি সেট হয়ে যাবে। (যেমন: git config --global user.name "নতুন নাম" )
  • মুছে ফেলা (Unsetting): কোনো নির্দিষ্ট সেটিং পুরোপুরি মুছে ফেলতে চাইলে লিখুন:
Bash
git config --global --unset user.name

৭. Troubleshooting (উইন্ডোজ ইউজারদের জন্য টিপস)

আপনি যেহেতু উইন্ডোজ ব্যবহার করছেন, কনফিগারেশন করার সময় কিছু বিষয়ে সতর্ক থাকবেন:

  • কোটেশন মার্ক: নাম দেওয়ার সময় সব সময় " " (Double Quotation) ব্যবহার করবেন, যেমন: "Kamrul Islam" । না দিলে গিট শুধু প্রথম অংশটুকু নিতে পারে।
  • কেস সেন্সিটিভিটি:ইমেইল সব সময় ছোট হাতের অক্ষরে দেওয়ার চেষ্টা করবেন।
  • এডিটর সমস্যা:যদি ভুল করে git config --global --edit লিখে ফেলেন এবং একটি কালো স্ক্রিন আসে যা থেকে বের হতে পারছেন না, তবে কিবোর্ডে :q! লিখে এন্টার দিন।
সারসংক্ষেপ:
  • ১. git config --global user.name "Name" (নাম সেট)
  • ২. git config --global user.email "Email" (ইমেইল সেট)
  • git config --list (সব চেক করা)

এভাবে একবার কনফিগার করে নিলে আপনার পিসির সব গিট প্রজেক্টে আপনার নাম ও ইমেইল অটোমেটিক চলে আসবে।

Lecture 5

Git Getting Started

Git-এ কাজ শুরু করার প্রসেসটি মূলত আপনার সাধারণ একটি ফোল্ডারকে গিটের "মস্তিষ্ক" বা নিয়ন্ত্রণে নিয়ে আসার প্রক্রিয়া।

নিচে প্রতিটি অংশ বিস্তারিতভাবে বুঝিয়ে বলা হলো:

১. Get Started with Git (শুরু করার প্রস্তুতি)

গিটে কাজ শুরু করার আগে নিশ্চিত করুন আপনার পিসিতে গিট ইন্সটল করা আছে। টার্মিনালে git --version লিখে এটি চেক করে নিন। এরপর প্রথম কাজ হলো গিটকে জানানো যে আপনি কে (Identity):

Bash
git config --global user.name "Your Name"
git config --global user.email "youremail@example.com"

২. Creating Git Folder (প্রজেক্ট ফোল্ডার তৈরি)

যেকোনো প্রজেক্ট শুরু করার জন্য প্রথমে একটি খালি জায়গা বা ফোল্ডার লাগে।

  • ম্যানুয়ালি: উইন্ডোজের সাধারণ ফোল্ডার যেভাবে তৈরি করেন।
  • টার্মিনালে: mkdir my-new-project লিখে এন্টার দিন। এরপর cd my-new-project লিখে ফোল্ডারের ভেতর প্রবেশ করুন।

৩. Initialize Git (গিট চালু করা)

ফোল্ডার তৈরি করলেই সেটি গিট প্রজেক্ট হয় না। গিটকে ওই ফোল্ডারের দায়িত্ব দেওয়ার জন্য আপনাকে এই ম্যাজিক কমান্ডটি দিতে হয়:

Bash
git init

এই কমান্ডটি দেওয়ার সাথে সাথে আপনার সাধারণ ফোল্ডারটি একটি Git Repository-তে রূপান্তরিত হয়।

৪. আপনার করা প্রশ্নগুলোর উত্তর

What is a Repository? (রিপোজিটরি কী?)

সহজ কথায়, রিপোজিটরি (বা সংক্ষেপে Repo) হলো আপনার প্রজেক্টের একটি ডিজিটাল স্টোরেজ বা ভান্ডার।

এটি সাধারণ ফোল্ডারের চেয়ে আলাদা। একটি সাধারণ ফোল্ডারে শুধু ফাইল থাকে, কিন্তু একটি রিপোজিটরিতে ফাইলের পাশাপাশি শুরু থেকে এ পর্যন্ত আপনি যতবার কোড পরিবর্তন করেছেন বা সেভ করেছেন, সেই সবগুলোর ইতিহাস (History) সংরক্ষিত থাকে।

What Happens When You Run git init?

যখন আপনি git init কমান্ডটি রান করেন, তখন পর্দার আড়ালে ৩টি প্রধান ঘটনা ঘটে:

  • .git ফোল্ডার তৈরি: আপনার প্রজেক্ট ফোল্ডারে একটি অদৃশ্য (hidden) ফোল্ডার তৈরি হয় যার নাম .git। আপনার সব হিস্ট্রি এবং ভার্সন এই ফোল্ডারের ভেতরেই জমা থাকে।

  • Tracking শুরু হওয়া: গিট ওই ফোল্ডারের সব ফাইলকে পর্যবেক্ষণ করা শুরু করে। আপনি এখন কোনো ফাইল তৈরি করলে গিট সেটি সাথে সাথে চিনতে পারবে।

  • Default Branch তৈরি: গিট স্বয়ংক্রিয়ভাবে একটি প্রাথমিক শাখা বা Branch তৈরি করে ফেলে (উইন্ডোজে সাধারণত যার নাম master দেখাবে)।

৫. Key Steps to Get Started (সফলভাবে শুরু করার মূল ধাপসমূহ)

একটি নতুন প্রজেক্ট গিটে শুরু করার সঠিক ক্রম হলো এটি:

  • git init: গিট চালু করুন।
  • ফাইল তৈরি: আপনার কোড বা ফাইল তৈরি করুন (যেমন: index.html)।
  • git add . : ফাইলগুলোকে স্টেজ বা লাইনে দাঁড় করান।
  • git commit -m "Initial commit" : ফাইলগুলো স্থায়ীভাবে সেভ করুন।

৬. Troubleshooting (সাধারণ সমস্যা ও সমাধান)

আপনি শুরু করার সময় নিচের সমস্যাগুলোতে পড়তে পারেন:

সমস্যা: .git ফোল্ডার খুঁজে পাচ্ছি না।
  • সমাধান: এটি ডিফল্টভাবে 'Hidden' থাকে। উইন্ডোজ ফাইল এক্সপ্লোরার থেকে View > Hidden items চেক করে দিলেই এটি দেখতে পাবেন।

>সমস্যা: git init দিয়েছি কিন্তু পরে মনে হলো দরকার নেই।
  • সমাধান: স্রেফ .git ফোল্ডারটি ডিলিট করে দিন। আপনার সাধারণ ফোল্ডারটি গিটের নিয়ন্ত্রণ থেকে মুক্ত হয়ে যাবে।

সমস্যা: ভুল ফোল্ডারে git init দিয়ে ফেলেছি।
  • সমাধান: ভয় নেই! আপনার ডেস্কটপ বা অন্য কোনো ভুল ফোল্ডারে যদি এটি দিয়ে ফেলেন, তবে ওই ফোল্ডারের ভেতর থাকা লুকানো .git ফোল্ডারটি ডিলিট করে দিলেই সব ঠিক হয়ে যাবে।

সমস্যা: কমান্ড প্রম্পটে (CMD) অদ্ভুত এরর দিচ্ছে।
  • অনেক সময় স্পেস বা টাইপিং মিস্টেক হয়। যেমন: gitinit না লিখে মাঝখানে একটি স্পেস দিয়ে git init লিখতে হবে।

একটি সহজ টিপ:

আপনি যেহেতু ইতোমধ্যে git reset নিয়ে কাজ করতে গিয়ে More? এররটি পেয়েছিলেন, তাই মনে রাখবেন—উইন্ডোজে গিট ব্যবহারের সময় কোনো স্পেশাল ক্যারেক্টার (যেমন: ^ বা ~) ব্যবহার করলে সেটিকে ডাবল কোটেশনের (" ") মধ্যে রাখা সবথেকে নিরাপদ।

Lecture 6

Git New Files

Git-এ কাজ করার সময় নতুন ফাইল তৈরি করা এবং সেগুলোকে গিটের নিয়ন্ত্রণে নিয়ে আসা খুবই গুরুত্বপূর্ণ। আপনি যখন একটি প্রজেক্টে নতুন কিছু যোগ করেন, গিট সেটাকে কীভাবে দেখে—সেই পুরো বিষয়টি নিচে বিস্তারিত আলোচনা করা হলো।

১. New Files সম্পর্কিত প্রশ্নের উত্তর

What is a New File? (নতুন ফাইল কী?)

একটি প্রজেক্ট ফোল্ডারে আপনি যখনই নতুন কোনো ফাইল তৈরি করেন (যেমন: about.html বা style.css), যা আগে সেখানে ছিল না, সেটিকেই গিট 'New File' হিসেবে গণ্য করে। তবে মনে রাখবেন, ফাইলটি ফোল্ডারে থাকলেই গিট সেটাকে সেভ করে না, আপনাকে গিটকে বলে দিতে হয় যে এটি গুরুত্বপূর্ণ।

Create a New File (নতুন ফাইল তৈরি)

আপনি উইন্ডোজ ফোল্ডারে রাইট ক্লিক করে ফাইল তৈরি করতে পারেন, অথবা সরাসরি টার্মিনাল (CMD/Bash) থেকে কমান্ড দিয়ে তৈরি করতে পারেন:

Bash
echo > index.html

(এটি index.html নামে একটি খালি ফাইল তৈরি করবে)

List Files in the Directory (ফাইল তালিকা দেখা)

আপনার বর্তমান ফোল্ডারে কী কী ফাইল আছে তা টার্মিনালে দেখতে নিচের কমান্ডটি ব্যবহার করুন:

Bash
dir

(Git Bash ব্যবহার করলে ls লিখতে পারেন)

Check File Status with git status

এটি গিটের সবথেকে প্রয়োজনীয় কমান্ড। ফাইলটি কোন অবস্থায় আছে (সেভ হয়েছে কি না, বা গিট চিনতে পারছে কি না) তা দেখার জন্য এটি ব্যবহার করা হয়:

Bash
git status

২. Tracked vs Untracked Files (সবথেকে গুরুত্বপূর্ণ অংশ)

গিট মূলত ফাইলগুলোকে দুটি প্রধান ভাগে ভাগ করে দেখে:

What is an Untracked File? (আনট্র্যাকড ফাইল কী?)

যখন আপনি একটি নতুন ফাইল তৈরি করেন, গিট সেটি দেখতে পায় কিন্তু সেটির দায়িত্ব নেয় না। গিট মনে করে এটি আপনার ব্যক্তিগত ফাইল হতে পারে যা প্রজেক্টের অংশ নাও হতে পারে।

  • git status দিলে এই ফাইলগুলো লাল রঙে দেখাবে।
  • গিট এই ফাইলের কোনো ব্যাকআপ বা হিস্ট্রি রাখে না যতক্ষণ না আপনি একে git add করছেন।

What is a Tracked File? (ট্র্যাকড ফাইল কী?)

যেসব ফাইল সম্পর্কে গিট আগে থেকেই জানে এবং যেগুলোর হিস্ট্রি সে রাখছে, সেগুলোই Tracked File।

  • আপনি যখন git add filename কমান্ডটি দেন, তখন একটি 'Untracked' ফাইল 'Tracked' ফাইলে পরিণত হয়।
  • এগুলো সাধারণত git status -এ সবুজ রঙে (Staged) দেখায়।

৩. কাজের ধাপ (Workflow)

নতুন ফাইল নিয়ে কাজ করার সময় আপনাকে এই চক্রটি অনুসরণ করতে হবে:

  • ফাইল তৈরি করুন: ( ফাইলটি এখন Untracked )।
  • git status দিন: গিট আপনাকে লাল রঙে ফাইলটি দেখাবে।
  • git add . দিন: এটি ফাইলটিকে Tracked করবে এবং Staging Area-তে নিয়ে যাবে।
  • git status দিন: এখন ফাইলটি সবুজ দেখাবে।
  • git commit -m "মেসেজ": এখন ফাইলটি স্থায়ীভাবে গিটের ইতিহাসে জমা হলো।

৪. Troubleshooting (সাধারণ সমস্যা ও সমাধান)

আপনি যখন নতুন ফাইল নিয়ে কাজ করবেন, কিছু সমস্যায় পড়তে পারেন:

সমস্যা: ফাইল তৈরি করেছি কিন্তু git status -এ আসছে না।
  • সমাধান: নিশ্চিত হয়ে নিন আপনি সঠিক ফোল্ডারে আছেন এবং ওই ফোল্ডারে git init করা আছে কি না।

সমস্যা: ফাইল ভুল করে add করে ফেলেছি, এখন আনট্র্যাকড করতে চাই।
  • সমাধান: লিখুন git rm --cached filename এতে ফাইলটি কম্পিউটার থেকে মুছবে না, শুধু গিটের লিস্ট থেকে বাদ যাবে।

সমস্যা: নতুন ফাইল তৈরি করার পর git commit -a দিয়েও কাজ হচ্ছে না।
  • সমাধান: মনে রাখবেন, নতুন বা Untracked ফাইলের ক্ষেত্রে -a কাজ করে না। আপনাকে অবশ্যই প্রথমবার git add হাতে করতে হবে।

সহজ কথা: নতুন ফাইল মানেই সেটি গিটের কাছে অপরিচিত (Untracked)। তাকে পরিচিত করতে git add করতে হয়, আর সেভ করতে git commit করতে হয়।

Lecture 7

Git Staging Environment

Git-এ কাজ করার সময় Staging Environment বা Staging Area-কে বলা হয় "কমিট করার আগের শেষ প্রস্তুতি"। আপনি যেহেতু উইন্ডোজ টার্মিনালে কাজ করছেন, তাই এই ধাপটি বোঝা আপনার জন্য খুবই জরুরি, কারণ এটি ছাড়া আপনি কোড সেভ বা কমিট করতে পারবেন না।

নিচে আপনার প্রশ্নগুলোর বিস্তারিত উত্তর দেওয়া হলো:

Here are some key commands for staging:

  • git add [file] - Stage a file
  • git add --all or git add -A - Stage all changes
  • git status - See what is staged
  • git restore --staged [file] - Unstage a file

১. Git Staging Environment কী?

সহজ কথায়, Staging Environment হলো একটি মাঝখানের জায়গা (Waiting Room)। আপনি আপনার ফাইলে অনেক কাজ করতে পারেন, কিন্তু তার সবটুকু হয়তো আপনি এখনই সেভ করতে চান না।

গিট আপনাকে সুযোগ দেয়—আপনার করা শত শত পরিবর্তনের মধ্য থেকে বেছে বেছে কিছু অংশ আলাদা করে রাখতে, যা আপনি পরের কমিটে যোগ করবেন। এই "বেছে রাখার জায়গাটিই" হলো স্টেজ।

২. ফাইল স্টেজ করা (git add)

আপনি যখন কোনো ফাইল তৈরি বা এডিট করেন, সেটি সরাসরি গিটের মেমোরিতে যায় না। তাকে স্টেজে পাঠানোর জন্য নিচের কমান্ডটি ব্যবহার করতে হয়:

Bash
git add filename.html

এটি দিলে নির্দিষ্ট ওই ফাইলটি স্টেজড (Staged) হয়ে যায়।

৩. একসাথে অনেক ফাইল স্টেজ করা

যদি আপনার প্রজেক্টে অনেকগুলো ফাইল থাকে এবং আপনি সবগুলোকে একসাথে স্টেজে পাঠাতে চান, তবে নিচের কমান্ডগুলো ব্যবহার করা হয়:

  • git add --all অথবা git add -A : এটি আপনার প্রজেক্টের সব নতুন ফাইল, এডিট করা ফাইল এবং ডিলিট করা ফাইল—সবকিছুকে একসাথে স্টেজ করে ফেলে।
  • git add .এটিও বর্তমান ফোল্ডারের সব ফাইল স্টেজ করে। এটিই সবথেকে বেশি ব্যবহৃত হয়।

৪. স্টেজড ফাইল চেক করা (git status)

আপনার ফাইলগুলো সফলভাবে স্টেজে গেছে কি না, তা চেক করার জন্য git status দিতে হয়।

  • ফাইলটি যদি লাল (Red) রঙে থাকে: এর মানে ফাইলটি এখনো Unstaged (স্টেজে যায়নি)।
  • ফাইলটি যদি সবুজ (Green) রঙে থাকে: এর মানে ফাইলটি Staged হয়ে গেছে এবং কমিট করার জন্য প্রস্তুত।

৫. ফাইল আনস্টেজ (Unstage) করা

কখনো কখনো ভুল করে কোনো ফাইল আপনি git add করে ফেলতে পারেন যা আপনি এই মুহূর্তে কমিট করতে চান না। ফাইলটি কম্পিউটার থেকে মুছবে না, শুধু স্টেজ থেকে সরিয়ে আগের অবস্থায় নিতে নিচের কমান্ডটি দিন:

Bash
git reset HEAD filename.html

( অথবা আধুনিক গিটে ব্যবহার করতে পারেন: git restore --staged filename.html )

৬. Troubleshooting (সাধারণ সমস্যা ও সমাধান)

আপনি স্টেজ করার সময় কিছু সমস্যায় পড়তে পারেন:

সমস্যা: git add . দিয়েছি কিন্তু কাজ হচ্ছে না।
  • সমাধান: নিশ্চিত হয়ে নিন আপনি সঠিক ফোল্ডারে আছেন এবং git init করা আছে।
সমস্যা: ফাইল এডিট করার পর আবার git status-এ লাল দেখাচ্ছে কেন?
  • সমাধান: আপনি একবার add করার পর যদি ফাইলে আবার নতুন কোনো কোড লিখেন, তবে গিট সেটি আবার লাল দেখাবে। আপনাকে আবারও git add করতে হবে নতুন পরিবর্তনটুকু যোগ করার জন্য।
সমস্যা: .gitignore ফাইল কাজ করছে না।
  • সমাধান: কোনো ফাইল যদি আগে থেকেই স্টেজড বা কমিটেড থাকে, তবে সেটি ইগনোর লিস্টে দিলেও লাভ হয় না। আগে তাকে git rm --cached দিয়ে স্টেজ থেকে সরাতে হয়।

একটি সহজ উদাহরণ দিয়ে পুরো বিষয়টি বুঝুন:

মনে করুন, আপনি অনেকগুলো ছবি তুলেছেন। সব ছবি ভালো হয়নি।

  • ১. ফোল্ডার: যেখানে সব ছবি আছে।
  • ২. Staging Area: আপনি বেছে বেছে সেরা ৫টি ছবি একটি আলাদা অ্যালবামে রাখার জন্য টেবিলে সাজালেন।
  • ৩. Commit: এবার আপনি নিশ্চিত হলেন এবং ছবিগুলো অ্যালবামে আঠা দিয়ে লাগিয়ে দিলেন।
  • মনে রাখবেন: কমিট করার আগে সব সময় git status দিয়ে দেখে নেবেন আপনার দরকারি সব ফাইল সবুজ রঙে (Staged) আছে কি না।

Lecture 8

Git Commit

গিট-এর হৃদপিণ্ড হলো এই Commit। আপনি যখন কোড লেখেন, তখন এক একটি নির্দিষ্ট কাজ শেষ করার পর সেটিকে স্থায়ীভাবে সেভ করে রাখাই হলো কমিট।

Here are some key commands for commits

  • git commit -m "message" - Commit staged changes with a message
  • git commit -a -m "message" - Commit all tracked changes (skip staging)
  • git log - See commit history

১. What is a Commit? (কমিট কী?)

সহজ কথায়, কমিট হলো আপনার প্রজেক্টের একটি স্ন্যাপশট (Snapshot) বা সেভ পয়েন্ট।

  • এটি আপনার ফাইলের বর্তমান অবস্থা গিটের ডাটাবেসে স্থায়ীভাবে জমা রাখে।
  • প্রতিটি কমিটের একটি ইউনিক আইডি (যেমন: e9997b2) থাকে, যা দিয়ে আপনি ভবিষ্যতে ওই অবস্থায় ফিরে যেতে পারেন।

২. How to Commit with a Message (-m)

সবথেকে বেশি ব্যবহৃত কমান্ড এটি। আপনি যখন কোনো ফাইল git add করে স্টেজিং এরিয়াতে আনেন, তখন সেটি সেভ করার জন্য মেসেজসহ কমান্ড দিতে হয়:

Bash
git commit -m  "Add login functionality"
  • কেন মেসেজ দরকার? মেসেজ দেখে আপনি বা অন্য ডেভেলপাররা বুঝতে পারেন ওই সময়ে কী পরিবর্তন করা হয়েছিল।

৩. Commit All Changes Without Staging (-a)

আপনি যদি ফাইলগুলো বারবার git add করতে না চান, তবে সরাসরি এই কমান্ডটি দিতে পারেন:

Bash
git commit -a -m  "Update all tracked files"
  • -a দিয়ে বোঝায় এটি একটি Annotated ট্যাগ।
  • -m দিয়ে সেই ভার্সন সম্পর্কে একটি ছোট মেসেজ যুক্ত করা হয়।

কাজ: এটি আপনার প্রজেক্টের যতগুলো ফাইল গিট আগে থেকেই চেনে (Tracked Files), সেগুলোর সব পরিবর্তন একসাথে স্টেজ করবে এবং কমিট করে দেবে।

৪. What happens if you try to commit a new file with -a?

এটি একটি কমন ভুল। আপনি যদি একদম নতুন একটি ফাইল তৈরি করেন (যা গিট এখনো চেনে না বা Untracked), তবে git commit -a কমান্ডটি কাজ করবে না।

  • কেন? গিট -a ফ্ল্যাগ ব্যবহার করলে শুধুমাত্র সেই ফাইলগুলোকেই দেখে যেগুলোকে সে আগে থেকেই ট্র্যাক করছে। নতুন ফাইলের ক্ষেত্রে আপনাকে প্রথমবার অবশ্যই git add ব্যবহার করতে হবে।

৫. Write Multi-line Commit Messages

কখনো কখনো আপনার কাজের বর্ণনা অনেক বড় হতে পারে। তখন শুধু একটি লাইনে মেসেজ না লিখে বিস্তারিত লিখতে পারেন:

  • টার্মিনালে শুধু git commit লিখে এন্টার দিন।
  • এটি আপনার পিসির ডিফল্ট এডিটর (যেমন: Notepad বা Vim) ওপেন করবে।
  • সেখানে প্রথম লাইনে হেডার লিখুন, এক লাইন গ্যাপ দিয়ে নিচে বিস্তারিত লিখে ফাইলটি সেভ করে ক্লোজ করে দিন।

৬. Commit Message Best Practices (সঠিক নিয়ম)

একটি প্রফেশনাল কমিট মেসেজ লেখার কিছু নিয়ম আছে:

  • সংক্ষিপ্ত রাখা: প্রথম লাইনটি ৫০ অক্ষরের মধ্যে রাখুন।
  • Present Tense ব্যবহার করা: "Fixed bug" না লিখে "Fix bug" লিখুন।
  • পরিষ্কার তথ্য: "Update" না লিখে "Change background color to blue" লিখুন।

৭. Other Useful Commit Options

--amend: সবশেষ কমিটটি সংশোধন করতে। (মেসেজ ভুল হলে বা কোনো ফাইল যোগ করতে ভুলে গেলে)।

Bash

--allow-empty: কোনো ফাইল পরিবর্তন না করেই একটি কমিট করতে (টেস্টিং এর জন্য)।

৮. Troubleshooting Common Commit Mistakes

  • ভুল ফাইল কমিট করা: যদি ভুল ফাইল সেভ করে ফেলেন, তবে git reset --soft "HEAD^" দিয়ে কমিটটি মুছে ফাইলগুলো ফিরিয়ে আনতে পারেন।
  • কমিট করতে ভুলে যাওয়া: অনেক কাজ একসাথে না জমিয়ে ছোট ছোট অংশে কমিট করার অভ্যাস করুন।
  • এডিটর থেকে বের হতে না পারা: যদি ভুলে শুধু git commit দিয়ে ভিম (Vim) এডিটরে ঢুকে যান, তবে কিবোর্ড থেকে :q! লিখে এন্টার দিয়ে বের হয়ে আসতে পারেন।

৯. View Commit History (git log)

আপনার করা সব কমিটের তালিকা দেখতে এটি ব্যবহার করা হয়:

  • বিস্তারিত দেখতে: git log
  • সংক্ষেপে এক লাইনে দেখতে: git log --oneline
  • কতগুলো ফাইল পরিবর্তন হয়েছে তা দেখতে: git log --stat

Git Tagging

Key Commands for Tagging

  • git tag [tagname] - Create a lightweight tag
  • git tag -a [tagname] -m "message" - Create an annotated tag
  • git tag [tagname] [commit-hash] - Tag a specific commit
  • git tag - List tags
  • git show [tagname] - Show tag details

গিট-এ কাজ করতে করতে যখন আপনার প্রজেক্ট একটি বিশেষ মাইলফলক বা গুরুত্বপূর্ণ পর্যায়ে পৌঁছায় (যেমন একটি নতুন ভার্সন রিলিজ), তখন সেই নির্দিষ্ট সময়টিকে একটি নাম দিয়ে চিহ্নিত করে রাখাই হলো Tagging। এটি অনেকটা বইয়ের পাতায় বুকমার্ক করে রাখার মতো।

নিচে প্রতিটি অংশ বিস্তারিতভাবে বুঝিয়ে দেওয়া হলো:

১. What is a Tag? (ট্যাগ কী?)

ট্যাগ হলো গিটের ইতিহাসের একটি নির্দিষ্ট পয়েন্টের জন্য একটি স্থায়ী লেবেল বা চিহ্ন। সাধারণত কোনো সফটওয়্যারের ভার্সন (যেমন: v1.0, v2.1) চিহ্নিত করতে এটি ব্যবহার করা হয়। এটি ব্রাঞ্চের (Branch) মতো পরিবর্তনশীল নয়; একবার ট্যাগ দিলে সেটি ওই নির্দিষ্ট কমিটেই (Commit) স্থির থাকে।

Some common tag types include:

  • Release Tags: সফটওয়্যারের ভার্সন (v1.0, v2.0) বোঝাতে।
  • Milestone Tags: প্রজেক্টের বড় কোনো ধাপ শেষ হলে।
  • Milestone Tags: কোডটি ব্যবহারের জন্য কতটা তৈরি তা বোঝাতে।

২. Annotated vs Lightweight Tags

গিট মূলত দুই ধরণের ট্যাগ ব্যবহার করে:

  • Lightweight Tag: এটি খুব সাধারণ। শুধুমাত্র একটি নির্দিষ্ট কমিটের নাম বা পয়েন্টার। এতে বাড়তি কোনো তথ্য (কে করেছে, কবে করেছে) থাকে না।

  • Annotated Tag: এটি গিটের ডাটাবেসে একটি পূর্ণাঙ্গ অবজেক্ট হিসেবে সেভ হয়। এতে ট্যাগের নাম ছাড়াও—কে ট্যাগ করেছে, কবে করেছে এবং একটি বর্ণনা (Message) থাকে। প্রফেশনাল কাজের জন্য এটিই সেরা।

৩. Key Commands for Tagging

ক. Create a Lightweight Tag

সহজ একটি ট্যাগ তৈরি করতে শুধু ট্যাগের নাম লিখুন:

Bash
git tag v1.0.0-lw

খ. Create an Annotated Tag (-a -m)

বিস্তারিত তথ্যসহ ট্যাগ তৈরি করতে এই কমান্ডটি ব্যবহার করুন:

Bash
git tag -a v1.0.0 -m "প্রথম অফিসিয়াল রিলিজ"
  • -a মানে Annotated এবং
  • -m দিয়ে মেসেজ যুক্ত করা হয়।

গ. Tag a Specific Commit

আপনি যদি আগে কোনো কমিট করে থাকেন যা তখন ট্যাগ করতে ভুলে গেছেন, তবে সেই কমিট আইডির (Checksum) মাধ্যমে ট্যাগ বসাতে পারেন:

Bash
git tag -a v1.2 9fceb02 -m "পুরনো কমিটের জন্য ট্যাগ"

ঘ. List Tags (ট্যাগের তালিকা দেখা)

আপনার প্রজেক্টে এ পর্যন্ত কতগুলো ট্যাগ আছে তা দেখতে:

Bash
git tag

ঙ. Show Tag Details (git show)

একটি নির্দিষ্ট ট্যাগের বিস্তারিত তথ্য (কে করেছে, কী মেসেজ) দেখতে:

Bash
git show v1.0.0

৪. Remote Operations (অনলাইনে ট্যাগ পাঠানো ও ডিলিট করা)

Push Tags to Remote

সাধারণ git push দিলে ট্যাগগুলো অনলাইনে (GitHub) যায় না। ট্যাগ পাঠাতে আলাদাভাবে কমান্ড দিতে হয়:

  • একটি নির্দিষ্ট ট্যাগ পাঠাতে: git push origin v1.0.0
  • সব ট্যাগ একসাথে পাঠাতে: git push origin --tags

Delete Tags

  • Local ডিলিট: git tag -d v1.0.0
  • Remote (GitHub) থেকে ডিলিট: git push origin --delete v1.0.0

Update or Replace a Tag (Force Push)

যদি ভুল কমিটে ট্যাগ দিয়ে ফেলেন এবং সেটি পরিবর্তন করতে চান, তবে আগের ট্যাগটি রিপ্লেস করতে হয়:

Bash
git tag -f v1.0.0 [new-commit-id]
git push origin --force --tags

৫. Tagging Best Practices

  • সব সময় Annotated Tag ব্যবহার করার চেষ্টা করুন কারণ এতে বিস্তারিত তথ্য থাকে।
  • ভার্সন কন্ট্রোলের জন্য Semantic Versioning (যেমন: v1.0.1) ফরম্যাট অনুসরণ করুন।
  • প্রতিটি বড় রিলিজের সময় একটি করে ট্যাগ দিয়ে রাখা ভালো।

Switch Statement - একাধিক Case

যখন একই variable এর অনেক values check করতে হয়, তখন switch ব্যবহার করা ভালো।

switch.js
let day = 4;
let dayName;
switch  (day) {    
    case 1: dayName = "শনিবার"; break;
    case 2: dayName = "রবিবার"; break;
    case 3: dayName = "সোমবার"; break;
    case 4: dayName = "মঙ্গলবার"; break;
    case 5: dayName = "বুধবার"; break;
    case 6: dayName = "বৃহস্পতিবার"; break;
    case 7: dayName = "শুক্রবার"; break;
    default : dayName = "ভুল দিন!";
}
        console.log (dayName); // "বুধবার"
    ✅ Switch এর গুরুত্বপূর্ণ অংশ:
  • case - প্রতিটা option
  • break - Switch থেকে বের হওয়া (না থাকলে পরের case ও চলবে!)
  • efault - কোনো case match না করলে (else এর মতো)
switch-multiple.js
let fruit = "আম";
let color;
switch  (fruit) {
    case "কলা" : case "আম" : color = "হলুদ"; break;
    case "আপেল" : case "স্ট্রবেরি" : color = "লাল"; break;
    case "কমলা" : color = "কমলা"; break;
    default : color = "জানি না";
}
        console.log ("রঙ:") + color); // "রঙ: হলুদ"

বিষয়                  If-Else                        Switch
ব্যবহার            যেকোনো condition           একই variable এর বিভিন্ন values
Comparison       Range check করা যায় (>, <)   শুধু equality (===)
Readability      অনেক conditions হলে জটিল   অনেক cases হলে সুন্দর
Performance      কম cases এ ভালো             বেশি cases এ faster
Multiple Conditions  &&, || ব্যবহার করা যায়      Multiple cases একসাথে
    🔑 Beginner Tips (খুব গুরুত্বপূর্ণ)
  • ✔️ Always use ===
  • ✔️ Code readable রাখো
  • ✔️ Nested if কম ব্যবহার করো
  • ✔️ Logic আগে ভাবো, পরে কোড লেখো

Logical Operators যুক্তি

একাধিক conditions একসাথে check করার জন্য logical operators ব্যবহার হয়।

তিনটা Logical Operators:

  • AND (&&)
  • OR (||)
  • NOT (!)
AND (&&)
সব condition true হতে হবে
if (age >= 18 && age <= 30) 

OR (||)
যেকোনো একটা true হলেই চলবে
if (age >= 18 || age > 60) 

NOT (!)
উল্টো করে দেয়- True হলে false রেজাল্ট দিবে আর false হলে true রেজাল্ট দিবে।
if (!isLoggedIn) 

// AND (&&) - দুইটাই সত্য হতে হবে
if (age >= 18 && hasTicket) {
    console.log ("সিনেমা দেখতে পারবে");
}

// OR (||) - যেকোনো একটা সত্য হলেই হবে
if (age < 12 || age > 60) {
    console.log ("ছাড় পাবে");
}

// NOT (!) - উল্টা করে
if (!hasTicket) {
    console.log ("টিকেট কিনতে হবে");
}
